# agents

Source-of-truth for the AGENTS.md instruction set used across projects.
Each `skills/<name>.md` is a self-contained, project-agnostic skill
definition that both Claude Code and OpenCode can read.
`bin/install-skills.sh` concatenates a selected subset into
`<target-project>/AGENTS.md` so the target tool has a single file to load.

In addition to those AGENTS.md skill fragments, this repo also ships
OpenCode TUI agent definitions under `opencode/agent/` — a separate
concept (interactive agents you cycle through with TAB), installed at
the user level by `bin/install-agents.sh`. See README.md for the
distinction.

This repo's own `AGENTS.md` describes the skill pipeline.
Project-specific conventions (for THIS repo) live in `docs/EXTRA.md`.

## Layout

```
skills/                       24 source skills — project-agnostic instructions
opencode/agent/               OpenCode TUI agent definitions (separate concept)
bin/install-skills.sh         concatenate selected skills → <project>/AGENTS.md
bin/check-skills.sh           list skills installed in a target project
bin/diff-skills.sh            report missing / stale / orphan skills in a project
bin/suggest-skills.sh         detect a project's stack, recommend skills
bin/install-agents.sh         install OpenCode TUI agents → user-level dir
bin/test/markers_test.sh      validate skill file structure
bin/test/install_smoke_test.sh  end-to-end install / diff / update cycle
docs/EXTRA.md                 this repo's own project-specific overrides
.github/workflows/ci.yml      prettier + shellcheck + smoke + structure tests
```

`bin/install-skills.sh` writes only inside the target project directory.
`bin/install-agents.sh` is the one exception — OpenCode loads agents
from a fixed user-level path, not per-project, so that script writes
to `~/.config/opencode/agent/`.

## Concatenation model

When `bin/install-skills.sh <project>` runs, the generated
`<project>/AGENTS.md` has three machine-readable regions:

```markdown
<!-- agents:banner:begin -->

> Auto-generated by `bin/install-skills.sh` ... defers to docs/EXTRA.md ...

<!-- agents:banner:end -->

<!-- agents:toc:begin -->

## Table of contents

- [Architect](#architect)
- ...
<!-- agents:toc:end -->

<!-- agents:begin name=architect sha=<sha256> -->

## Architect

... body ...

<!-- agents:end name=architect -->

<!-- agents:begin name=rails sha=<sha256> -->

## Rails

...

<!-- agents:end name=rails -->
```

Each marker is an HTML comment — invisible to a Markdown reader, but
greppable by `bin/diff-skills.sh` and `bin/check-skills.sh`. The
marker namespace stays `agents:` (rather than `skills:`) because the
output file is named AGENTS.md and any project that installed under
the old script naming continues to work with the new scripts. The
`sha=` on each begin marker is the sha256 of the source skill file at
install time; `diff-skills.sh` compares it against the current source
to detect stale blocks.

## Workflow

Daily use:

```bash
# Discover what's recommended for a project
bin/suggest-skills.sh ~/Dev/my-rails-app

# Install everything (or a focused subset)
bin/install-skills.sh ~/Dev/my-rails-app --include rails,postgres,reviewer --with-extra-stub

# Inspect what's installed
bin/check-skills.sh ~/Dev/my-rails-app

# After editing skills/<name>.md here, re-sync the target project
bin/install-skills.sh ~/Dev/my-rails-app --mode update

# Drift check
bin/diff-skills.sh ~/Dev/my-rails-app
```

Install modes:

- **update** (default) — replace existing skill blocks with current
  source, add any new skills from `--include`, regenerate the banner
  and TOC. Leaves any non-marker content (custom preamble) untouched.
- **append** — only add skills not already present. Never overwrites
  an existing skill block.
- **override** — rebuild `<project>/AGENTS.md` from scratch. Destructive.

## Skill file shape

Every `skills/*.md` follows the same template so concatenation produces
a coherent AGENTS.md, and so a reader can predict where to find each
kind of content:

```markdown
---
name: <skill-name> # must match the filename (sans .md)
description: <one-liner>
triggers: [<patterns suggest-skills.sh keys on>]
---

# <Title>

## Project context

Read `docs/EXTRA.md` first. ...

## Conventions

- ...

## Anti-patterns

- ...

## Commands / verification

- ...
```

The "Project context" section is the contract that keeps generic
skills useful across projects: anything declared in `docs/EXTRA.md`
wins over defaults stated in the skill body.

## docs/EXTRA.md

Each target project has its own `docs/EXTRA.md` — `install-skills.sh
--with-extra-stub` scaffolds a starter. That file holds:

- Project-wide hard rules that don't belong in the generic skills.
- Per-skill overrides (e.g., "for `rails`, the view layer is
  ViewComponent, not ERB").
- Pointers to other docs the skills should read (architecture notes,
  spec directories, runbooks).

`install-skills.sh` never edits `docs/EXTRA.md` after the initial
stub. The project owns it.

## Conventions for editing this repo

- One skill per file under `skills/`. Filename = name in frontmatter.
- All four required sections must be present (the markers_test.sh
  enforces this).
- Markdown wraps at ~80 cols for prose (prettier enforced in CI).
- Bash scripts: `#!/usr/bin/env bash`, `set -euo pipefail`, shellcheck-
  clean.
- Source skill files are project-agnostic. Anything project-specific
  belongs in the target project's `docs/EXTRA.md`, NOT in the skill
  source.
- Don't push without running `bin/test/markers_test.sh` and
  `bin/test/install_smoke_test.sh` locally — CI runs them, but a green
  PR review here propagates a broken skill to every project that
  installs from this repo.

## CI

`.github/workflows/ci.yml` runs on every push and PR:

- `prettier --check '**/*.md'` — markdown formatting.
- `shellcheck bin/**/*.sh` — bash lint.
- `bash -n` on every shell script — syntax check.
- `bin/test/markers_test.sh` — every skill file passes structure checks.
- `bin/test/install_smoke_test.sh` — full install/diff/update cycle.

## Hard rules

1. Source files stay project-agnostic. Project-specific conventions
   belong in the target project's `docs/EXTRA.md`, not in `skills/`.
2. `bin/install-skills.sh` writes only inside the target project.
   Never `~/.claude`, `~/.codewhale`, `~/.deepseek`, or any user-level
   path. (`bin/install-agents.sh` is the one user-level installer in
   this repo — see its header for scope.)
3. Don't hand-edit a project's generated AGENTS.md long-term — edit
   the source here and re-run `install-skills.sh`. The exception is
   the non-marker preamble, which `update` mode preserves.
4. Don't push skill edits without re-reading the diff. A broken skill
   propagates to every project that installs from this repo.

## Glossary

- **skill** — one `.md` file under `skills/`, describing a coherent
  topic or role. Concatenated into a target project's AGENTS.md.
- **agent** — an OpenCode TUI agent definition under `opencode/agent/`.
  Installed user-level by `bin/install-agents.sh`. Distinct from
  "skill" — agents are interactive TUI roles, skills are instruction
  fragments for an LLM tool.
- **target project** — any repo where you want skills installed via
  `bin/install-skills.sh <path>`.
- **marker** — HTML-comment delimiters (`<!-- agents:begin name=... -->`,
  `<!-- agents:end ... -->`) wrapping each skill's block in the
  generated AGENTS.md. Marker namespace is `agents:` (matches the
  output filename, kept stable for back-compat).
- **EXTRA** — `docs/EXTRA.md` in a target project; holds the
  project-specific overrides the generic skills defer to.
